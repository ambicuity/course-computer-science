## The Problem

In Lesson 02 you saw that classical ciphers fail because the key space is too small or the algorithm leaks statistical patterns about the plaintext. The one-time pad (OTP) fixes both problems at once: if the key is truly random, as long as the message, and never reused, then every ciphertext is equally likely for every plaintext — perfect secrecy, proved by Shannon in 1949.

The catch? Generating, distributing, and protecting keys as long as every message is impractical for almost every real system. The moment you reuse a key, the perfect secrecy proof collapses and you get a "two-time pad" — an attacker who intercepts two ciphertexts can XOR them together to cancel the key and recover the XOR of the two plaintexts, which leaks enormous structure.

Stream ciphers are the engineering answer: instead of a random key as long as the message, use a short key to seed a pseudorandom number generator (PRNG) whose output *looks* random. XOR the PRNG's output (the keystream) with the plaintext. This is efficient, but the security now rests entirely on the quality of the PRNG. If the PRNG is predictable (LFSR), biased (RC4), or malleable, the cipher breaks.

This lesson builds both the broken designs (LFSR, RC4) and the modern fix (ChaCha20) so you understand exactly where each fails and why.

## The Concept

### One-Time Pad: perfect secrecy

The OTP encrypts a plaintext *P* by XOR with a uniformly random key *K* of the same length:

```
C = P ⊕ K
```

Decryption is identical — XOR the ciphertext with the same key:

```
P = C ⊕ K
```

Shannon's 1949 theorem: if K is uniformly random, then for any ciphertext C and any plaintext P, `Pr[C | P] = 1/|message_space|`. Every plaintext is equally likely given the ciphertext. The attacker learns *nothing*.

Three conditions that must all hold:
1. **Key is truly random** — not pseudorandom, not a counter, not a hash output.
2. **Key is as long as the message** — no short key expanded via PRNG.
3. **Key is never reused** — not even once.

Violate any one and the proof collapses.

### Two-time pad: key reuse catastrophe

If `C₁ = P₁ ⊕ K` and `C₂ = P₂ ⊕ K` (same key K reused):

```
C₁ ⊕ C₂ = (P₁ ⊕ K) ⊕ (P₂ ⊕ K) = P₁ ⊕ P₂
```

The key cancels. The attacker gets P₁ ⊕ P₂, which is a deterministic function of the two plaintexts. If either plaintext is known (or guessed), the other is recovered instantly. Even when neither is known, the XOR of two natural-language texts reveals word boundaries, repeating patterns, and is easily decrypted with crib-dragging.

### Stream ciphers: OTP with a PRNG

A stream cipher replaces the truly random key K with a keystream Z generated by a deterministic function:

```
Z = PRNG(seed = short_key || nonce)
C = P ⊕ Z
```

The keystream Z must be indistinguishable from random to any polynomial-time adversary. If it isn't, the cipher leaks information about P.

### LFSR: the simplest (broken) stream cipher

A **Linear Feedback Shift Register** of degree *n* is an *n*-bit state register combined with a linear recurrence defined by taps (a polynomial over GF(2)):

```
s[i+n] = c₁·s[i+n-1] ⊕ c₂·s[i+n-2] ⊕ ... ⊕ cₙ·s[i]
```

If the taps correspond to a **primitive polynomial** over GF(2), the LFSR cycles through all 2ⁿ − 1 nonzero states before repeating — a maximal-length sequence (m-sequence).

LFSRs are fast, simple, and completely broken as stream ciphers. The **Berlekamp-Massey algorithm** takes any LFSR output sequence and recovers the minimal polynomial after observing at most 2n consecutive bits. Once the polynomial is known, the entire future (and past) keystream is determined.

### RC4: fast, biased, dead

RC4 (Rivest Cipher 4) was designed in 1987 and used in SSL/TLS and WEP for decades. It has two phases:

1. **KSA (Key Scheduling Algorithm):** initialize a 256-byte permutation S using the key.
2. **PRGA (Pseudo-Random Generation Algorithm):** repeatedly swap and output bytes from S.

RC4 has critical flaws:
- The **first bytes of keystream are biased**: the second byte is zero with probability 2/256 instead of 1/256 (the Fluhrer-Mantin-Shamir attack exploited this to break WEP).
- **Long-term statistical biases** in the keystream allow plaintext recovery given enough ciphertext.
- RC4 was formally prohibited in RFC 7465 (2015).

### ChaCha20: the modern answer

ChaCha20, designed by Daniel J. Bernstein, is a stream cipher built from an ARX (Add-Rotate-XOR) core. It takes a 256-bit key, 96-bit nonce, and 32-bit counter, and produces a keystream block by block.

The heart is the **quarter round** operating on four 32-bit words (a, b, c, d):

```
a += b;  d ^= a;  d <<<= 16
c += d;  b ^= c;  b <<<= 12
a += b;  d ^= a;  d <<<= 8
c += d;  b ^= c;  b <<<= 7
```

The state is a 4×4 matrix of 32-bit words:

```
"expa"  "nd 3"  "2-by"  "te k"     ← constants
key₀    key₁    key₂    key₃       ← key (words 0-3)
key₄    key₅    key₆    key₇       ← key (words 4-7)
ctr₀    nonce₀   nonce₁  nonce₂     ← counter + nonce
```

Twenty rounds of quarter rounds (10 double-rounds of column then diagonal rounds) produce one 64-byte keystream block. ChaCha20 is used in TLS 1.3, SSH, WireGuard, and the Linux kernel's `/dev/urandom`.

### CTR mode: block cipher → stream cipher

Counter mode turns any block cipher (like AES) into a stream cipher:

```
keystream_block_i = E_K(counter + i)
C = P ⊕ keystream_0 ‖ keystream_1 ‖ ...
```

CTR mode is **parallelizable** (each block is independent) and **seekable** (decrypt block N without computing blocks 0..N−1). It provides no authentication — always use it with a MAC (or use AES-GCM directly).

## Use It

### Production stream ciphers

**ChaCha20-Poly1305** (the AEAD variant) is the default cipher in:
- **TLS 1.3** (RFC 8446, §5.1) — mandatory-to-implement alongside AES-GCM.
- **WireGuard** — uses ChaCha20-Poly1305 for all tunnel encryption.
- **SSH** (RFC 8709) — `chacha20-poly1305@openssh.com`.
- **Linux kernel** — `chacha20_block()` in `drivers/char/random.c` for `/dev/urandom`.

The Rust `chacha20` crate (part of the RustCrypto project) provides a production implementation. Compare it with the one in `code/main.rs` — the production version adds SIMD acceleration, zeroization of key material, and constant-time operations to resist side channels.

**AES-CTR** is the other major stream-cipher construction. OpenSSL implements it in `crypto/modes/ctr128.c`.

### RC4 in the wild (historical)

RC4 was formally prohibited in TLS by RFC 7465. WEP's use of RC4 was broken by the Fluhrer-Mantin-Shamir attack (2001), which exploited the key-dependent biases in the KSA. This is why you should never design your own cipher and why stream ciphers need more than a PRNG — they need a PRNG that is indistinguishable from random under adversarial analysis.

## Key Terms

| Term | What people say | What it actually means |
|------|----------------|------------------------|
| One-Time Pad | "Unbreakable encryption" | A provably secure cipher with three impossible requirements: random key, key as long as message, never reuse the key |
| Two-time pad | "Reusing a one-time pad" | The catastrophic failure mode where XORing two ciphertexts encrypted with the same key cancels the key, revealing P₁ ⊕ P₂ |
| LFSR | "A simple random number generator" | A linear recurrence over GF(2) whose output is completely predictable from 2n consecutive bits via Berlekamp-Massey |
| Stream cipher | "OTP but practical" | XOR plaintext with a keystream generated by a PRNG seeded from a short key — security depends entirely on the PRNG's pseudorandomness |
| ChaCha20 quarter round | "The mixing function" | Four ARX operations (add, rotate, XOR) on four 32-bit words that provide diffusion; applied 20 times total in 10 double-rounds |
| CTR mode | "Turning AES into a stream cipher" | Encrypt successive counter values with a block cipher to produce a keystream; parallelizable and seekable but provides no authentication |
| Berlekamp-Massey | "LFSR solver" | An algorithm that recovers the minimal LFSR feedback polynomial from as few as 2n output bits, breaking any LFSR-based stream cipher |

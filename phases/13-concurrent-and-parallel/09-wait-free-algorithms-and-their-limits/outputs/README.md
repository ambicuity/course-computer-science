# Wait-Free Algorithms and Their Limits — Outputs

## Artifacts

This directory contains the reusable artifacts produced by Phase 13, Lesson 09.

### Reference Implementation: `../code/main.rs`

Three implementations live in `main.rs`:

| Artifact | Type | Description |
|----------|------|-------------|
| `WaitFreeCounter` | Wait-free | Atomic counter using `fetch_add`. Every call completes in exactly 1 hardware instruction (`LOCK XADD` on x86). No thread can starve. |
| `AtomicSnapshot` | Wait-free | Snapshot of N registers using the double-collect technique. Consistent point-in-time reads with bounded retry (at most N+1 iterations). |
| `TreiberStack` | Lock-free (not wait-free) | Classic lock-free stack using CAS. Demonstrates starvation under contention: some threads may retry indefinitely while others make progress. |

### Correctness Properties Demonstrated

- **Wait-Free Counter:** All N threads complete exactly `OPS_PER_THREAD` (100k) increments. Variance = 0.
- **Wait-Free Snapshot:** No torn reads. Every scan returns a consistent state that existed at some instant, even under concurrent writers.
- **Lock-Free Stack:** Some threads complete fewer operations than others. The min/mean/max distribution is skewed under contention.

### Key Measurements

Run with:
```
rustc ../code/main.rs && ./main
```

Expected output pattern (8 threads, 100k ops each):
- Wait-Free Counter: all threads show exactly 100,000 ops.
- Lock-Free Stack: min thread shows significantly fewer ops.

### When to Use Each

| Situation | Choice |
|-----------|--------|
| Soft real-time (audio, games, trading) | Wait-free (bounded steps) |
| High-throughput server (many requests) | Lock-free (CAS retry loops are fine) |
| Mixed: reads must be fast, writes can wait | Lock-free reads + mutex writes (Go's sync.Map pattern) |
| N registers that must be read consistently | Wait-free snapshot (double-collect) |
